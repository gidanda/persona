# Persona Relationship Layer 拡張概要

作成日: 2026-03-17

## 1. 位置づけ

本書は、既存の Persona MVP に対して追加を検討している `Relationship Layer` 機能群を整理した設計メモである。

正式な要件定義書そのものではなく、以下を明文化するための参照文書として扱う。

- 追加機能の背景と目的
- プロダクトコンセプトの再定義
- MVP に含める範囲
- 将来機能の候補
- 実装方針の叩き台
- 情報構造と概念構造

## 2. 現状の MVP

既存の Persona MVP は以下の 3 つの利用導線を中心に構成されている。

- `People`: スキャン済みユーザー一覧と他者プロフィール閲覧
- `Scan`: QR コードによるプロフィール交換
- `Me`: 自分のプロフィール編集と表示

現時点の価値の中心は、プロフィールを作る、交換する、見る、の 3 点にある。一方で、会った後に関係を前進させる仕組みはまだ弱い。

## 3. 追加機能の目的

### 3.1 背景

Persona が単なるプロフィール共有アプリに留まると、デジタル名刺やプロフィール帳に近い体験になりやすい。

本来狙うべき価値は、リアルで会った相手との関係を前進させることにある。

### 3.2 目指す流れ

```text
プロフィールを見る
↓
相手の興味やカルチャーに反応する
↓
2人の間に Shared Topic が生まれる
↓
必要に応じてその話題について会話する
↓
関係が前進する
```

### 3.3 コンセプト

Persona の拡張後の中核は次のように定義する。

> Persona は、人との関係を共有された話題で育てるアプリである。

操作レベルでは、次のように言い換えられる。

> プロフィールは土台であり、その上に `Reaction -> Shared -> Thread` の関係レイヤーを追加する。

## 4. 基本思想

### 4.1 人中心

Persona はあくまで人中心のサービスとする。

- 主役は人との関係
- 媒介として音楽、映画、本、場所、興味対象を扱う
- 結果として Shared Topic と会話が生まれる

コンテンツ自体を探索する SNS にはしない。

### 4.2 利用タイミング

利用の中心は次の場面とする。

- 会った後
- 継続的に相手の変化や興味を見るとき
- 相手のプロフィール更新をきっかけに関係が動くとき

### 4.3 チャットを主役にしない

Persona は汎用 DM アプリを目指さない。

会話は常設チャットではなく、`Shared Topic` に紐づく小さなスレッドとして扱う。

## 5. 追加後のアプリ構造

### 5.1 Bottom Navigation

`Shared` を主要機能としてボトムナビゲーションに追加する。

```text
People | Scan | Shared | Me
```

### 5.2 各画面の役割

- `People`: 知っている人のプロフィールを見る入口
- `Scan`: QR で交換する入口
- `Shared`: 自分と他者の間に生まれた共有話題と関係を見る入口
- `Me`: 自分のプロフィール管理

## 6. MVP で追加する機能

### 6.1 Reaction

他人のプロフィール内の各項目に対して、軽い反応を起こせるようにする。

候補アクション例:

- 共感
- 気になる
- 興味あり

重要なのは最初の行動を軽くすること。長文メッセージやギフト送信ではなく、軽いリアクションから関係が始まる設計にする。

### 6.2 Shared

Reaction や推薦などの共有行為によって、2 人の間に `Shared Topic` を自動生成する。

`Shared` は関係の中心画面であり、「誰との間に、どんな話題が生まれているか」を見る場所とする。

### 6.3 Thread

`Shared Topic` ごとに小さな会話スレッドを持てるようにする。

ただし、スレッドは Shared 作成と同時には開始せず、どちらかが最初のテキストを送った時に開始されるものとする。

この設計により、次の状態を両立する。

- Shared は軽く増える
- 空のチャット乱立を防げる
- 本当に話したい話題だけが会話化する

### 6.4 People からの導線

People からユーザーを開いた際に、プロフィール画面から Shared へ遷移できる導線を持たせる。

UI 案:

- `Profile | Shared` タブ
- `View Shared` ボタン

重要なのは、プロフィールから Shared に自然に移動できること。

## 7. 情報構造

### 7.1 基本構造

```text
User
↓
Relationship
↓
Shared Topic
↓
Thread Message
```

### 7.2 Relationship

2 人のユーザー間の関係を表す単位。

例:

- User A × User B

`Shared Topic` は必ず Relationship に属する。

### 7.3 Shared Topic

2 人の間で共有された 1 つの話題や対象。

例:

- Radiohead
- Past Lives
- 建築
- 渋谷のカフェ

プロフィール項目、Recent、レコメンドなどを起点として生まれる。

### 7.4 Thread

`Shared Topic` に紐づいたテキスト会話。

Shared が作られた時点では Thread は始まっておらず、最初のテキスト送信時に開始される。

## 8. Shared Topic の成立条件

### 8.1 自動生成タイミング

`Shared Topic` は、次のような行為が発生した時点で自動生成される。

- プロフィール項目への共感
- プロフィール項目への興味表明
- Recent への反応
- レコメンド送信

### 8.2 Thread の開始条件

```text
Shared Topic 自動生成
↓
どちらかが最初のテキスト送信
↓
Thread 開始
```

この分離は設計上重要である。

- `Shared` は関係の素材
- `Thread` はその素材に言葉が乗った状態

## 9. Shared Topic が持つべき情報

MVP 時点で `Shared Topic` に持たせたい主な情報は以下。

### 9.1 参加者

- 自分
- 相手

### 9.2 共有対象

- 曲
- 映画
- 本
- 場所
- 興味テーマ
- その他プロフィール項目

### 9.3 発生理由

`origin` の例:

- `empathy`
- `interest`
- `recommendation`
- `reaction_to_recent`

### 9.4 発生元コンテキスト

`source_context` の例:

- `profile_music`
- `profile_movie`
- `profile_book`
- `recent`
- `direct_share`

### 9.5 状態

最低限、以下の状態を想定する。

- `passive_shared`: Shared に存在するが会話未開始
- `active_thread`: 会話中
- `archived`: 長期間動いていない

## 10. 同一話題の扱い

同じ 2 人の間で同じ対象が複数回扱われる場合、基本方針は次の通り。

> 同じ 2 人 × 同じコンテンツは、原則 1 つの Shared Topic に統合する。

例:

- 最初は共感で Shared に入る
- 後でおすすめも行う
- さらに会話が発生する

これらを別トピックで乱立させず、1 つの Shared Topic の履歴として蓄積する方が自然である。

## 11. プロフィール項目と外部コンテンツ

### 11.1 基本方針

プロフィールに載せる音楽、映画、本、場所などは、単なる文字列入力ではなく、可能な範囲で実コンテンツと紐づける方向を推奨する。

### 11.2 段階的実装

MVP:

- 手入力または簡易検索による登録
- タイトル、作者、URL 程度を内部表現として保持
- Shared / Reaction の対象として使えることを優先

将来:

- Spotify、Apple Music、地図、映画 DB、書籍 DB などの外部サービスと連携
- コンテンツ ID や URL を保持
- 遷移、プレビュー、再生などを可能にする

### 11.3 設計上の考え方

最初から全ジャンルで深い外部連携を行うと実装コストが高い。

MVP では次を優先する。

- 実在コンテンツに近いデータ構造を持つ
- アプリ内の関係促進ロジックを先に成立させる
- 深い外部 API 連携は後続で段階的に実装する

## 12. UI 導線

### 12.1 基本フロー

```text
People
↓
他人のプロフィールを見る
↓
Reaction
↓
Shared Topic が生まれる
↓
Shared でその人との共有話題を見る
↓
必要なら Thread を開始する
```

### 12.2 Shared 画面の役割

Shared 画面は、アプリ全体で最も重要な機能の 1 つになる。

想定導線:

```text
Shared
↓
アクションが発生したユーザー一覧
↓
User A
↓
A との Shared 一覧
↓
Topic 詳細
↓
Thread
```

### 12.3 MVP での見せ方

MVP では、Shared は単純な一覧でよい。

例:

- Radiohead
- Past Lives
- 建築

補助表示候補:

- 共感で追加
- Recent 由来
- 会話あり / なし
- 最終更新日時

## 13. Home 機能

`Home` は将来追加したい機能として検討するが、今回の MVP 必須機能には含めない。

### 13.1 役割

Home の役割は、SNS のようなコンテンツ消費ではなく、プロフィールを見に行くきっかけを作ることと定義する。

### 13.2 現時点の方向性

- 更新された人のプロフィールカードが流れる
- カードには誰がどのジャンルを更新したかをアイコンで表示する
- タップで該当プロフィールやセクションへ遷移する

例:

- `Ken 🎵📚` = Ken が音楽と本を更新

### 13.3 注意点

Home をイベントフィードにしすぎると、人中心設計からズレる。

そのため次を方針とする。

- 更新内容そのものを細かく流しすぎない
- 人カード単位で束ねる
- プロフィールへの導線として使う

## 14. 将来機能

以下は今回の MVP には含めず、将来機能として整理する。

### 14.1 Together / Experienced

実際に一緒に体験したものを Shared の延長レイヤーとして扱う案。

例:

- 一緒に観た映画
- 一緒に聴いた曲
- 一緒に行った場所

体験登録フロー、写真管理、状態遷移の追加設計が必要なため後続とする。

### 14.2 アルバム

Together / Experienced に紐づく写真アルバム。

### 14.3 レコメンド高度化

「この人に合いそうなものを送る」体験を強化する。

### 14.4 AI 補助

将来的には以下を検討できる。

- 相手に合いそうなコンテンツ提案
- 話題になりやすい項目の提示
- 共通点抽出

MVP 時点では不要。

## 15. 実装方針

### 15.1 最小実装の原則

既存の `People / Scan / Me` 構造を壊さずに、Relationship Layer を 1 枚追加する考え方で進める。

### 15.2 実装優先順位

1. Profile 項目への Reaction
2. Shared の生成と一覧表示
3. Shared Topic 詳細画面
4. Thread の開始と表示
5. Home
6. Together / Experienced

### 15.3 実装上の注意

- Shared と Thread を分ける
- 同じ話題は統合する
- 空 Thread を作りすぎない
- チャットアプリ化しない
- Shared を関係の中心として扱う

## 16. この拡張による変化

追加前:

```text
プロフィールを作る
↓
交換する
↓
見る
```

追加後:

```text
プロフィールを作る
↓
交換する
↓
見る
↓
反応する
↓
共有話題が生まれる
↓
必要に応じて話す
↓
関係が前進する
```

Persona はこの追加によって、単なるプロフィールアプリから、会った人との関係を共有話題で前進させるアプリへ進化する。

## 17. 現時点の MVP 定義

### 17.1 含めるもの

- Bottom Navigation への `Shared` 追加
- プロフィール項目への Reaction
- Shared Topic 自動生成
- Shared 一覧
- Shared Topic ごとの Thread
- Thread は最初のテキスト送信で開始

### 17.2 含めないもの

- Home 本実装
- Together / Experienced
- アルバム
- AI recommendation
- 高度な外部サービス連携

## 18. 一文での定義

> Persona における追加機能は、プロフィール上の興味対象への反応をきっかけに、2 人の間に Shared Topic を生み、その話題ごとに必要な会話を行える Relationship Layer を追加するものである。
