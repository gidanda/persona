UX上の重要ルール 2. DB設計 RDB（PostgreSQL / Supabase /
Prismaを想定）を前提とし、MVPではプロフィール本体・SNSリンク・Taste・交換・フォローの5領域を明確に分ける。
2-1. テーブル一覧 2-2. users
プロフィールの中核となるテーブル。QR遷移先のURL安定化のため、display_name
とは別に username を持つ前提とする。 2-3. social_links
SNSリンクを可変個数で持つためのテーブル。users
に直書きせず分離することで拡張しやすくする。 2-4. taste_entries
Tasteカード用のテーブル。ユーザーはカテゴリ候補から最大3件まで登録できる。
2-5. exchanges
QRを読み取った時点で交換が成立することを記録するテーブル。MVPでは「誰が誰を読み取ったか」を片方向で保存する。
MVPでは UNIQUE(from_user_id, to_user_id)
を置き、同一方向の重複交換を防ぐ。 2-6. follows
交換した相手の中から、よく見返したい人を整理するための軽量な関係テーブル。重いSNS的フォローではなく、People画面の上段整理用として扱う。
MVPでは UNIQUE(user_id, target_user_id) を置き、重複フォローを防ぐ。
2-7. 関係図（概念） ・users 1 - n social_links ・users 1 - n
taste_entries ・users n - n users （exchanges を介して交換関係を表現）
・users n - n users （follows を介して継続閲覧対象を表現） 2-8.
